# 0.8.4 — streaming reader: `readRowGroups()` iterator

## What
New public reader API:

```java
Iterator<RowGroup> LastraReader#readRowGroups()
```

with a companion `RowGroup` value type exposing:

```java
int rowCount()
long tsMin()
long tsMax()
int index()
long[]   getLongColumn(int colIdx)
double[] getDoubleColumn(int colIdx)
byte[][] getBinaryColumn(int colIdx)
```

Columns are decoded **lazily on first access**. When the consumer calls
`Iterator#next()`, the previous group's decoded arrays are released
(`RowGroup.clear()`), so heap residency stays bounded to one row group
of decompressed data plus the per-codec scratch state — regardless of
how many groups the file contains.

For files written in the legacy flat layout (no explicit row groups),
the iterator emits a single synthetic group covering the whole file.
That synthetic group's columns still go through the existing full-file
decoder (there is no smaller streaming unit available without re-writing
the file), but the API shape stays uniform so downstream converters can
use one code path.

## Why
The existing `readSeriesLong / readSeriesDouble / readSeriesBinary` family
materialises full-size primitive arrays for every column it returns. Used
by `LastraToParquetConverter` and `LastraToCsvConverter`, that pattern OOMs
under `-Xmx512m` on real datasets: a 158 MB Lastra file from
`2025-W41.parquet` (11.6M rows, 6 DOUBLE columns) needs roughly
`6 × 11.6M × 8 B ≈ 530 MB` just for the decoded double[] arrays, before
codec scratch buffers or the parquet writer's row dehydrator allocate
anything. The reverse-path converter then throws
`OutOfMemoryError: Java heap space` at
`LastraReader.readSeriesBinary(LastraReader.java:429)` — the symmetric of
the forward-path OOM that `lastra-convert` 0.14.0 fixed.

`readRowGroups()` is the read-side analogue of the streaming
`writeSeries(rgRows, columns)` pattern that landed in 0.8.3: each step
yields exactly one writer's row group at a time, and the consumer is in
control of when to drop the decoded buffers.

## How
- New `RowGroup` class. Holds the per-RG stats (rowCount/tsMin/tsMax),
  the column kinds, and a lazy decoded-array cache. `clear()` nulls
  the cache so it can be GC'd.
- New iterators inside `LastraReader`:
  - `MultiRowGroupIterator` — for files with explicit row groups; on
    `next()` it calls `clear()` on the previously yielded group, then
    builds the next one. Reuses the private `extractRgColumn(...)`
    helper that was already there for `readRowGroupLong/Double/Binary`.
  - `SingleRowGroupIterator` — one-shot iterator that emits a synthetic
    whole-file group for the no-row-groups layout. Columns route through
    the existing `readSeriesLong/Double/Binary` so behaviour matches the
    legacy reader exactly.
- Three package-private helpers on `LastraReader`
  (`decodeRowGroupLong / Double / Binary`) so `RowGroup` can ask for one
  column's decoded array without going through the public name-based API.

## Compat
- File format: unchanged. Reads every existing Lastra file.
- Existing public APIs (`readSeriesLong / Double / Binary`,
  `readRowGroupLong / Double / Binary`, `rowGroupStats`, etc) are
  unchanged. Callers using them keep working.

## Validation
- 28/28 unit tests pass (was 26 in 0.8.3; +2 for the new iterator:
  `testReadRowGroupsIterator` and `testReadRowGroupsIteratorFlatLayout`).
- End-to-end test in `lastra-convert` 0.14.1 (companion release):
  `2025-W41.lastra` (158 MB, 1151 row groups) → `roundtrip.parquet`
  under `-Xmx512m`. Previously OOM'd on
  `LastraReader.readSeriesBinary(LastraReader.java:429)`. Now succeeds.
