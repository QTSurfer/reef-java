# lastra-java 0.8.1 — idempotent close + legacy 4-byte trailer reader

## Summary

Two fixes that together unbreak the qtsurfer hourly Lastra → Redis publisher
pipeline, which was failing with `ArrayIndexOutOfBoundsException` on most
frate instruments per run on 2026-04-26.

### Writer: `LastraWriter.close()` is now idempotent

Earlier callers (notably `lastra-convert ≤ 0.12.0`'s
`LastraFratesHourlyBackfill`, `LastraKlinesHourlyBackfill`,
`LastraHourlyBackfill`) called `w.close()` explicitly **inside** a
try-with-resources block, so close() ran twice. Each call rebuilt and emitted
the full file body, so every backfill output was written **doubled** —
two complete Lastra blobs concatenated in one file. Production
`/mnt/data/lastra-backfill/` was 2× expected size.

`close()` now no-ops after the first invocation. Test:
`LastraRoundtripTest.testCloseIsIdempotent`.

### Reader: handle the legacy 4-byte FOOTER_MAGIC trailer

The footer-size hint added at `f92828c` (added after the 0.8.0 tag) made the
trailer 8 bytes — `[FOOTER_MAGIC][footerSize]`. Files written by writers
that predate the hint (lastra-java ≤ 0.8.0, lastra-convert ≤ 0.12.0) only
have the 4-byte magic. The HEAD reader's row-groups branch was
unconditionally reading `data.length - 4` as footerSize; on legacy files
that's the magic itself (≈ 1.28e9), making `footerPos` wildly negative and
throwing `ArrayIndexOutOfBoundsException` at construction time.

Added `LastraReader.scanRowGroupsForFooter`: forward-walks the data section
in groups of `seriesColCount` length-prefixed columns and finds the footer
position by matching the remaining-bytes budget against the closed-form
footer size for the current `rgCount`. Test:
`LastraRoundtripTest.testRowGroupsWithLegacyFourByteTrailer`.

## Migration

Consumers should bump `<lastra.version>` to `0.8.1`. Re-running existing
backfill cronjobs once with `--skip-existing` removed will rewrite all
duplicated files via the new idempotent writer.
